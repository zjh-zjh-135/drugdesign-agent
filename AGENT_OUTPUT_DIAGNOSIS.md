# Agent 输出体验问题诊断与优化方案

## 问题复现

### 问题1：一键全流程返回质量极差
**用户输入：** `帮我跑一遍AKT1的全流程`

**实际返回：**
```
全流程完成。项目ID=42，生成50个分子，通过8个，获得Top 3候选分子。
```

**问题：** 用户输入了靶点、等待了数分钟，最终只得到一句话摘要。没有分子结构、没有性质数据、没有排名对比。这等于“白跑了”。

---

### 问题2：后续追问直接失败
**用户输入：** `给出详细的这三个分子数据`

**实际返回：**
```
执行步骤
Step 1: LLM 规划失败，先获取项目状态
Action: get_project_status({"project_id":42})
Status: ok
Step 2: 基于状态获取建议
Action: suggest_next_step({"project_id":42})
Status: ok
执行报告
成功
```

**问题：** 用户要的是数据，Agent 莫名其妙去“获取状态”和“建议下一步”，没有返回任何分子数据。这是最致命的——**Agent 不理解用户在问什么**。

---

### 问题3：无输入时自动触发（Copilot 自主执行）
**用户没有输入**，系统却自动执行 `get_project_status` + `suggest_next_step`，输出“下一步建议”（空内容）。

**问题：** 这是前端 Copilot 的自动轮询逻辑，用户在等待，系统在空转。这是非常糟糕的体验。

---

## 根因分析

| 根因 | 说明 |
|------|------|
| **1. 输出格式化严重缺失** | `run_full_pipeline` 返回的是原始 JSON 字典。引擎没有专门的“格式化器”，直接丢了 `message` 字段给用户。用户的`top_molecules` 数组里有完整数据，但从未被渲染成可读报告。 |
| **2. 上下文断链** | 用户说“**这**三个分子数据”时，Agent 应该意识到是指上一步（ID=42）的结果。但当前系统没有“将前一次工具结果作为上下文进行格式化输出”的能力。 |
| **3. suggest_next_step 被默认触发** | 当 LLM 规划失败或意图模糊时，系统默认调用 `suggest_next_step`，这个工具本身只是返回空提示，没有实际价值。 |
| **4. 前端空消息轮询** | Copilot 组件可能在用户无输入时自动发送空消息到 `/agent/execute`，导致不必要的后端调用。 |
| **5. 缺少“结果解释”专用能力** | 系统有生成工具、有分析工具，但没有“把结果翻译成人话”的工具。 |

---

## 优化方案（按优先级排序）

### 方案A：立即见效——为 `run_full_pipeline` 添加内置格式化（最高优先级）

**做法：** 在 `run_full_pipeline` 返回时，如果 `top_molecules` 有数据，自动生成一份专业 Markdown 报告，直接放在 `final_report` 字段中。

**报告格式示例：**
```markdown
## AKT1 全流程完成 — 候选分子报告

**项目信息：** ID=42 | 靶点：AKT1 | PDB：4EKL
**执行结果：** 生成50个分子，通过8个，进入 Top 3 候选：

### 🥇 候选分子 #1（综合得分：82.5）
- **SMILES：** `CC(C)N1CCC(Oc2ccc3c(c2)CCN(C2CCC2)C3=O)CC1`
- **分子量：** 342.5 | **LogP：** 3.2 | **QED：** 0.85
- **对接分数：** -9.5 | **ADMET得分：** 78 | **毒性：** 低风险
- **与已知活性分子对比：** 与 MK-2206 结构骨架相似，但引入了环丙基降低 hERG 风险

### 🥈 候选分子 #2（综合得分：79.2）
...

### 🥉 候选分子 #3（综合得分：76.8）
...

### 💡 下一步建议
- 候选分子#1 QED 和 ADMET 表现最佳，建议优先合成验证
- 分子#2 对接分数略低，但合成可及性评分更高（SA Score 3.2），可作为备选
```

**为什么优先：** 用户的核心痛点是“跑了但没拿到可读的报告”。这是数据到信息的最后一道坎，不解决这个，前面的 Pipeline 都是白跑。

---

### 方案B：修复“追问格式化”——新增 `format_top_molecules` 工具

**做法：** 注册一个专门工具，接收 `project_id` 和 `limit`，返回格式化的分子报告。

**触发场景：**
- 用户说“详细数据”→ 意图解析检测到 `follow_up` + 没有明确动作 + 上下文有 `project_id` → 自动调用 `format_top_molecules(project_id=42)`
- 或者直接在 `run_full_pipeline` 的 `final_report` 中预生成，用户问时直接从记忆中取

**为什么：** 解决“上下文断链”问题。用户追问是对上一个结果的深入，不是请求新操作。

---

### 方案C：移除 `suggest_next_step` 的兜底触发

**做法：**
1. 在 `engine.py` 中，当 LLM 规划失败时，**不再默认调用 `suggest_next_step`**
2. 改为返回诚实提示："抱歉，我没有理解您的需求。请告诉我您想查看分子数据、分析失败原因，还是调整参数重新运行？"
3. 或者，如果上下文有项目ID，直接返回 `format_top_molecules` 的结果作为兜底

**为什么：** `suggest_next_step` 返回的是“空建议”，用户看了等于没看。诚实的澄清比虚假的“成功”更好。

---

### 方案D：修复前端空消息轮询

**做法：**
1. 检查 `HelpChatModal` 或 Copilot 组件，是否有定时器自动发送空消息
2. 在前端发送消息前，增加 `if (!message.trim()) return;` 的空消息拦截
3. 后端 `/agent/execute` 入口也增加空消息校验：`if not user_message.strip(): return {"success": False, "error": "消息不能为空"}`

**为什么：** 用户没说话，系统不应该有任何动作。这是最基本的产品体验。

---

### 方案E：为“追问”设计专门的意图解析规则

**做法：** 在 `intent_parser.py` 中新增规则：

```python
# 如果用户问的是“详细数据/详细结果/给我看看/具体分析”等
# 且上下文有 project_id，但消息中没有明确动作 → 判定为格式化追问
follow_up_format_patterns = ["详细数据", "详细结果", "具体数据", "给我看看", "详细点", "展开说说"]
if any(p in msg_lower for p in follow_up_format_patterns) and project_id:
    return ParsedIntent(
        primary_type=IntentType.FOLLOW_UP,
        detected_actions=["format_top_molecules"],  # 不再是 suggest_next_step
        suggested_tools=["format_top_molecules"],
    )
```

**为什么：** 让 Agent 知道用户不是在要求“新操作”，而是在要求“把已有的结果格式化给我看”。

---

## 实施建议

| 优先级 | 方案 | 工作量 | 影响 |
|--------|------|--------|------|
| P0 | 方案A（内置格式化） | 1小时 | 核心痛点，立即见效 |
| P0 | 方案D（空消息拦截） | 10分钟 | 阻止无意义触发 |
| P1 | 方案C（移除兜底） | 20分钟 | 减少虚假“成功” |
| P1 | 方案B（格式化工具） | 30分钟 | 解决追问场景 |
| P2 | 方案E（追问意图） | 30分钟 | 体验更智能 |

**建议立即执行 P0 + P1，这四个方案加起来不到 2 小时，但能直接解决 80% 的体验问题。**
